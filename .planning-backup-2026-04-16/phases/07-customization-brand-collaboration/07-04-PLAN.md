# Plan 07-04: Brand Portal Frontend + Customization Flow Completion + POD Integration

## Objective

Complete the customization flow with payment integration for custom orders, add POD (Print-on-Demand) service provider abstraction, and create a basic brand management portal web page. Wire the end-to-end flow from design through payment.

## Context

@07-CONTEXT.md -- decisions D-07 through D-12
@apps/backend/src/modules/customization/customization.service.ts
@apps/backend/src/modules/payment/ -- existing payment module

## Threat Model

| File | Surface | Mitigation |
|------|---------|------------|
| pod-integration.ts | external API call | timeout, retry, validate response |
| customization payment | payment flow | verify quote ownership, amount match |

## Tasks

### Task 1: Add POD Service Provider Abstraction

<task type="auto">
<read_first>
- apps/backend/src/modules/customization/customization.service.ts
</read_first>
<action>
1. Create `apps/backend/src/modules/customization/pod/pod-provider.interface.ts`:
   - Interface `PODProvider` with methods:
     - `submitOrder(designData, template, shippingAddress): Promise<PODOrderResult>`
     - `getOrderStatus(providerOrderId): Promise<PODOrderStatus>`
     - `getAvailableProducts(): Promise<PODProduct[]>`
   - Types: PODOrderResult (providerOrderId, estimatedDelivery, status), PODOrderStatus (status, trackingNumber, carrier), PODProduct (type, name, availableColors, availableSizes)

2. Create `apps/backend/src/modules/customization/pod/mock-pod-provider.ts`:
   - MockPODProvider implementing PODProvider
   - Returns simulated responses for development
   - Generates random tracking numbers
   - Simulates 3-day production time

3. Create `apps/backend/src/modules/customization/pod/pod-service.ts`:
   - PODService that resolves to configured provider (mock by default)
   - `submitToProduction(customizationRequestId)` -- after payment confirmed, submit design to POD provider
   - `checkProductionStatus(customizationRequestId)` -- poll POD status
   - Configuration via environment variable POD_PROVIDER (default: 'mock')
</action>
<acceptance_criteria>
- PODProvider interface defines submit, status, products
- MockPODProvider returns valid simulated data
- PODService resolves provider by config
- submitToProduction creates order with provider
- checkProductionStatus returns current status
</acceptance_criteria>
</task>

### Task 2: Add Customization Payment + Production Flow

<task type="auto">
<read_first>
- apps/backend/src/modules/customization/customization.service.ts
- apps/backend/src/modules/payment/ (for payment integration patterns)
</read_first>
<action>
1. Extend `CustomizationService` with:
   - `confirmAndPay(requestId, userId, paymentMethod)` -- validate quote exists, create payment order via PaymentService, update status to 'confirmed'
   - `handlePaymentCallback(requestId, paymentResult)` -- webhook handler, on success update status and submit to POD
   - `updateProductionStatus(requestId, status, trackingInfo?)` -- update request status during production
   - `completeOrder(requestId)` -- mark as completed when delivered
   - Add payment tracking fields to CustomizationRequest via schema: paymentId, podOrderId, trackingNumber, carrier

2. Add new status values if needed:
   - Current: draft, submitted, quoting, confirmed, in_progress, completed, cancelled
   - These are sufficient -- 'in_progress' maps to production phase
   - Add 'shipped' between in_progress and completed

3. Add to CustomizationController:
   - `POST /customization/:id/pay` -- initiate payment for confirmed customization
   - `GET /customization/:id/production-status` -- check production/shipping status
   - `POST /customization/:id/confirm-delivery` -- user confirms delivery

4. Update schema.prisma:
   - Add `paymentId String?` to CustomizationRequest
   - Add `podOrderId String?` to CustomizationRequest
   - Add `trackingNumber String?` to CustomizationRequest
   - Add `carrier String?` to CustomizationRequest
   - Add `shipped` to CustomizationStatus enum
</action>
<acceptance_criteria>
- confirmAndPay validates quote and creates payment
- Payment callback triggers POD submission
- Production status tracking works
- New controller endpoints registered
- Schema updated with payment/tracking fields
</acceptance_criteria>
</task>

### Task 3: Add Packaging + Notification Integration for Customization

<task type="auto">
<read_first>
- apps/backend/src/modules/customization/customization.service.ts
- apps/backend/src/modules/notification/ (for notification patterns)
</read_first>
<action>
1. Add packaging configuration to customization flow:
   - In `CustomizationService`, after POD order submitted, create packaging record
   - Packaging includes: AiNeed branded box, thank you card, brand sticker
   - This is metadata on the customization request (packagingNotes field)

2. Add notification triggers for customization status changes:
   - `CUS_QUOTE_READY` -- quote calculated, notify user to review
   - `CUS_PAYMENT_CONFIRMED` -- payment successful, production started
   - `CUS_PRODUCTION_STARTED` -- POD provider accepted order
   - `CUS_SHIPPED` -- order shipped with tracking number
   - `CUS_DELIVERED` -- order delivered
   - Use existing notification module patterns
   - If notification module has event-based pattern, emit events
   - If not, add TODO comments for future integration

3. Add estimated delivery calculation:
   - Base: estimatedDays from quote + 2 days shipping
   - Store as estimatedDeliveryDate on CustomizationRequest
</action>
<acceptance_criteria>
- Packaging metadata added to customization flow
- 5 notification trigger points defined
- Estimated delivery date calculated
- Status change hooks in service methods
</acceptance_criteria>
</task>
