# Pitfalls Research — 代码规整陷阱

**Project:** 寻裳 (XunO) 代码规整
**Researched:** 2026-04-16

## 陷阱 1: 在无测试保障下大规模重构

**严重度**: 🔴 致命
**检测信号**: 后端 15%、移动端 5% 测试覆盖率，无 CI 测试门禁

**问题**: 低测试覆盖率意味着重构后无法验证功能是否被破坏。寻裳 35+ 后端模块、50+ 移动端页面，任何重构都可能在不知不觉中引入回归。

**预防策略**:
1. 每个域规整前，先为该域的关键路径补充集成测试
2. 建立 CI 测试门禁（PR 必须通过测试）
3. 使用 `nest build` + `tsc --noEmit` 作为最低保障
4. 对移动端使用截图测试（Detox）覆盖核心流程

**应处理阶段**: Phase 0（准备阶段）

## 陷阱 2: 循环依赖重构导致运行时崩溃

**严重度**: 🔴 致命
**检测信号**: 16 个文件使用 forwardRef，最严重的是 AiStylistModule ↔ RecommendationsModule

**问题**: 消除循环依赖需要移动服务或引入事件驱动解耦。如果移动不当，NestJS 的依赖注入会在运行时抛出异常，而不是编译时报错。

**预防策略**:
1. 使用 dependency-cruiser 可视化当前依赖关系
2. 先添加事件驱动接口，再逐步移动服务
3. 每次只解一个循环依赖，立即运行测试验证
4. 将 Recommendations 移至 platform 层作为共享服务，域间通过事件解耦

**应处理阶段**: Phase 2（后端域划分）

## 陷阱 3: 设计系统迁移破坏 UI 一致性

**严重度**: 🟠 高
**检测信号**: 778 处硬编码颜色、921 处硬编码 fontSize、971 处硬编码间距

**问题**: codemod 批量替换可能将语义不同的相同值错误合并。例如 `#333` 可能是文字颜色也可能是边框颜色，替换为同一个 Token 会导致语义丢失。

**预防策略**:
1. 先审计所有硬编码值的使用上下文，按语义分类
2. 创建语义化 Token（如 `text.primary` 而非 `gray.900`）
3. codemod 按语义分类替换，而非按值替换
4. 每次替换后视觉回归测试
5. 保留原始值作为注释，便于回溯

**应处理阶段**: Phase 1（设计系统统一）

## 陷阱 4: Polyfill 修改引发连锁崩溃

**严重度**: 🟠 高
**检测信号**: 13 个 polyfill 文件被 100+ 文件引用

**问题**: polyfill 是整个移动端的依赖基础。修改或移除任何一个 polyfill 都可能导致 100+ 文件编译失败。

**预防策略**:
1. Polyfill 规整放在最后阶段
2. 逐个替换为真实依赖，每次替换后全量编译
3. 保留 polyfill 作为 fallback，新代码使用真实依赖
4. 不批量修改，一次只动一个 polyfill

**应处理阶段**: Phase 4（移动端重组后期）

---

## 陷阱 5: 范围蔓延——规整变成重写

**严重度**: 🟠 高
**检测信号**: 开始"顺便"重构业务逻辑、修改 API 接口、调整数据库模型

**问题**: 规整项目最容易失控。每次打开一个文件，都会发现"可以顺便改一下"的地方。这些小改动累积起来就变成了重写，风险和工期都失控。

**预防策略**:
1. 明确定义"规整"边界：只改结构/命名/类型，不改业务逻辑
2. 每个 Phase 的 PLAN.md 必须列出"不改什么"
3. Code Review 时检查是否有业务逻辑变更
4. 使用 `git diff --stat` 监控每个 Phase 的变更量

**应处理阶段**: 所有阶段

---

## 陷阱 6: 后端模块域划分破坏事件驱动架构

**严重度**: 🟡 中
**检测信号**: 4 个事件监听器（payment→subscription, payment→notification, order→notification, subscription→payment）创建跨模块耦合

**问题**: 域划分后，事件监听器可能跨越域边界。如果将 payment 和 subscription 分到不同域，事件监听器需要重新设计。

**预防策略**:
1. 保留事件驱动架构，域间通过事件通信
2. 将事件定义放在 common 层
3. 监听器放在消费方域内
4. 使用 dependency-cruiser 验证域间无直接依赖

**应处理阶段**: Phase 2（后端域划分）

---

## 陷阱 7: 移动端导航重组破坏深层链接

**严重度**: 🟡 中
**检测信号**: 59 个文件包含导航调用，30+ 深层链接路由，33 个守卫路由

**问题**: 移动端导航系统复杂（5 tab + 50+ stack screen + 深层链接 + 路由守卫）。重组页面结构时，路由名称和参数可能变化，导致深层链接失效。

**预防策略**:
1. 导航类型定义集中管理（已有 navigation/types.ts）
2. 页面重组时保持路由名称不变
3. 使用路由映射层解耦页面组件和路由名
4. 深层链接测试覆盖

**应处理阶段**: Phase 3（移动端重组）

---

## 陷阱 8: PII 加密在重构中意外暴露

**严重度**: 🔴 致命
**检测信号**: 18 个文件涉及 PII 加密，user-key.service.ts 有 TypeScript 错误

**问题**: 移动加密相关代码时，如果遗漏加密步骤，敏感数据可能以明文存储。这在 GDPR/PIPL 合规下是严重违规。

**预防策略**:
1. PII 加密代码标记为"不可移动"
2. 加密相关文件在域划分时归入 identity 域
3. 添加数据审计日志，检测明文 PII
4. 安全 Review 作为每个 Phase 的必要步骤

**应处理阶段**: Phase 2（后端域划分，identity 域优先）

---

## 陷阱 9: 共享包（@xuno/types, @xuno/shared）形同虚设

**严重度**: 🟡 中
**检测信号**: @xuno/types 在移动端仅 1 处引用，后端 0 处；@xuno/shared 类似

**问题**: 共享包存在但几乎不被使用，各应用自己定义类型。规整时如果不激活共享包，monorepo 的类型一致性优势无法发挥。

**预防策略**:
1. 后端域划分时，将跨域共享类型提取到 @xuno/types
2. 移动端重组时，将 API 响应类型提取到 @xuno/types
3. 建立"类型优先从共享包导入"的规范
4. ESLint 规则禁止在应用内重复定义共享类型

**应处理阶段**: Phase 2-3

---

## 陷阱 10: Recommendations 模块过度工程化

**严重度**: 🟡 中
**检测信号**: 25+ 服务文件，3 个子模块，GNN/Transformer/SASRec 等高级算法

**问题**: Recommendations 模块包含 25+ 服务，远超实际需要。规整时如果试图全部保留，维护负担不降反升。

**预防策略**:
1. 审计哪些推荐算法实际在生产中使用
2. 未使用的算法标记为 @deprecated
3. 保留核心推荐管道，移除未使用的算法实现
4. 将 Recommendations 降级为 platform 层共享服务

**应处理阶段**: Phase 2（后端域划分，platform 层）
