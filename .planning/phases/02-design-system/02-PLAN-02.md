# Plan 02: 颜色 Token 迁移 Codemod

## Objective

将所有硬编码颜色值（~972 hex + ~271 rgba）迁移至 DesignTokens.colors.* 语义化 Token。按语义上下文分类替换，避免相同值不同语义的错误合并。

## Tasks

### Task 02-02: 创建颜色迁移 Codemod

<objective>
编写 ts-morph codemod 脚本，根据语义映射表将硬编码颜色值替换为 DesignTokens 引用。按语义上下文智能替换，避免错误合并。
</objective>

<read_first>
- apps/mobile/scripts/semantic-color-map.json
- apps/mobile/src/theme/tokens/design-tokens.ts
- apps/mobile/src/theme/tokens/colors.ts
- apps/mobile/src/theme/index.ts
</read_first>

<action>
1. 创建 `apps/mobile/scripts/codemod-colors.ts`：

```typescript
// 使用 ts-morph 解析 TSX/TS 文件
// 逻辑:
// 1. 加载 semantic-color-map.json
// 2. 遍历 src/ 下所有 .tsx/.ts 文件（排除 node_modules, theme/tokens/）
// 3. 在 StyleSheet.create() 和内联样式中查找颜色属性
// 4. 根据属性名推断语义上下文:
//    - color/text*Color → text 上下文
//    - background*/bg → background 上下文
//    - border*Color → border 上下文
// 5. 在映射表中查找替换 Token
// 6. 替换硬编码值为 DesignTokens.colors.xxx 引用
// 7. 如果文件已有 import { DesignTokens } from '...'，复用；否则添加 import
// 8. 保留原始值作为注释: /* was: #1A1A2E */
// 9. 输出替换报告: { total, replaced, skipped, byFile }
//
// 特殊处理:
// - rgba() 值: 如果匹配映射表则替换，否则保留（如玻璃效果）
// - Colors.xxx 引用: 替换为对应的 DesignTokens.colors.xxx
// - theme.colors.xxx 引用: 已由 Plan 01 修复，保留
// - 渐变色: 保留，不在本阶段处理
```

2. 在 `apps/mobile/package.json` 中添加脚本：

```json
{
  "scripts": {
    "codemod:colors": "npx ts-node scripts/codemod-colors.ts",
    "codemod:colors:dry-run": "npx ts-node scripts/codemod-colors.ts --dry-run"
  }
}
```

3. Codemod 必须支持 `--dry-run` 模式，只输出报告不修改文件。
4. Codemod 必须支持 `--file <path>` 模式，只处理指定文件。
</action>

<acceptance_criteria>
- `apps/mobile/scripts/codemod-colors.ts` 文件存在
- `apps/mobile/package.json` 包含 `codemod:colors` 和 `codemod:colors:dry-run` 脚本
- `cd apps/mobile && npx ts-node scripts/codemod-colors.ts --dry-run` 输出替换报告
- dry-run 模式不修改任何文件
- 替换报告中 `replaced` 数 > 0
</acceptance_criteria>

<verification>
运行 dry-run 模式，确认报告合理。然后对单个文件运行 codemod，验证替换正确。
</verification>

### Task 02-03: 执行颜色迁移 Codemod

<objective>
运行颜色迁移 codemod，将所有硬编码颜色值替换为 DesignTokens 引用。分批执行，每批后验证编译。
</objective>

<read_first>
- apps/mobile/scripts/codemod-colors.ts
- apps/mobile/scripts/semantic-color-map.json
</read_first>

<action>
1. 先运行 dry-run 获取替换报告：`cd apps/mobile && npx ts-node scripts/codemod-colors.ts --dry-run > scripts/color-migration-report.json`

2. 审查报告中的 `skipped` 项（无法自动映射的值），决定手动处理方式

3. 执行 codemod：`cd apps/mobile && npx ts-node scripts/codemod-colors.ts`

4. 执行后运行类型检查：`cd apps/mobile && npx tsc --noEmit`

5. 修复类型错误（如有）

6. 处理 Colors.xxx 引用的迁移（89 文件, ~886 处）：
   - `Colors.primary[500]` → `DesignTokens.colors.brand.primary`
   - `Colors.primary[400]` → `DesignTokens.colors.brand.secondary`
   - `Colors.neutral[200]` → `DesignTokens.colors.backgrounds.secondary`
   - `Colors.neutral[100]` → `DesignTokens.colors.backgrounds.tertiary`
   - `Colors.neutral[800]` → `DesignTokens.colors.text.primary`
   - `Colors.neutral[600]` → `DesignTokens.colors.text.secondary`
   - `Colors.white` → `DesignTokens.colors.text.inverse`（text 上下文）或 `DesignTokens.colors.backgrounds.primary`（bg 上下文）

7. 运行审计脚本验证：`cd apps/mobile && npx ts-node scripts/audit-hardcoded-colors.ts`
</action>

<acceptance_criteria>
- `cd apps/mobile && npx tsc --noEmit` 无新增类型错误
- `cd apps/mobile && npx ts-node scripts/audit-hardcoded-colors.ts` 报告用户可见组件中硬编码颜色 < 50 处（从 ~972 降低）
- 所有 `Colors.primary[xxx]` 和 `Colors.neutral[xxx]` 引用已替换为 DesignTokens
- 替换后的文件包含 `import { DesignTokens } from` 导入语句
- 原始值保留为注释 `/* was: #xxx */`
</acceptance_criteria>

<verification>
运行审计脚本和类型检查，确认迁移进度。
</verification>

### Task 02-04: 手动处理无法自动映射的颜色值

<objective>
处理 codemod 跳过的硬编码颜色值（rgba 玻璃效果、渐变色、特殊用途颜色等），确保用户可见组件中 0 处硬编码颜色。
</objective>

<read_first>
- apps/mobile/scripts/color-migration-report.json
- apps/mobile/src/theme/tokens/design-tokens.ts
</read_first>

<action>
1. 读取 color-migration-report.json 中的 `skipped` 列表

2. 对每个跳过的值进行分类处理：
   - **rgba 玻璃效果**（如 `rgba(255,255,255,0.8)`）：添加到 DesignTokens.colors.backgrounds（如 `glassLight`, `glassDark`）
   - **渐变色值**：保留在 StyleSheet 中，但添加 `// gradient-todo` 注释标记
   - **特殊用途颜色**（如 `transparent`）：替换为 `'transparent'` 常量
   - **季节性颜色**：替换为 `DesignTokens.seasonal.xxx` 或 `SeasonColors.xxx`

3. 在 `design-tokens.ts` 中补充缺失的 token 定义：

```typescript
// 在 DesignTokens.colors.backgrounds 中添加
glassLight: 'rgba(255, 255, 255, 0.8)',
glassMedium: 'rgba(255, 255, 255, 0.6)',
glassDark: 'rgba(0, 0, 0, 0.5)',
glassDarker: 'rgba(0, 0, 0, 0.3)',
```

4. 在 `darkTokens` 中添加对应的暗色模式值

5. 手动替换剩余硬编码值
</action>

- `cd apps/mobile && npx ts-node scripts/audit-hardcoded-colors.ts` 报告用户可见组件中硬编码颜色 = 0 处
- `design-tokens.ts` 包含 `glassLight`, `glassMedium`, `glassDark`, `glassDarker` token
- `darkTokens` 包含对应的暗色模式值
- 所有跳过的值已处理或标记为 `// gradient-todo`
</acceptance_criteria>

<verification>
运行审计脚本，确认硬编码颜色为 0。
