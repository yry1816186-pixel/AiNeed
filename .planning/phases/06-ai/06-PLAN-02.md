<objective>

**T-06-04: 认证绕过** — 合并后确保所有端点仍受 APIKeyMiddleware 保护。无新增认证边界变更。
Severity: low — 路由合并，安全边界不变。
</threat_model>

</read_first>
<action>
创建 `ml/api/routes/stylist.py`，合并两个路由文件的所有功能。设计原则：

1. **统一前缀**: `/api/stylist`
2. **统一 Schema**: 使用 `ml/api/schemas/stylist.py` 中的已有 Schema，补充缺失的
3. **统一响应格式**: 所有端点返回 `{success: bool, data: ..., timestamp: str}` 格式
4. **保留所有端点**: 合并两个文件的所有端点，v2 独有端点直接纳入

路由设计：
```
POST /api/stylist/chat              — 对话（来自 v1 + v2）
POST /api/stylist/outfit            — 穿搭推荐（来自 v1 + v2，异步任务）
POST /api/stylist/analyze-body      — 体型分析（来自 v1 + v2）
GET  /api/stylist/conversation/{id} — 获取对话历史（来自 v1）
DELETE /api/stylist/conversation/{id} — 清除对话历史（来自 v1）
GET  /api/stylist/fashion-trends/{season} — 时尚趋势（来自 v2）
GET  /api/stylist/color-guide/{season} — 色彩指南（来自 v2）
GET  /api/stylist/body-type-guide/{type} — 体型指南（来自 v2）
GET  /api/stylist/occasion-guide/{occasion} — 场合指南（来自 v2）
GET  /api/stylist/health            — 健康检查（来自 v2）
```

实现要点：
- 使用 `from ml.services.intelligent_stylist_service import ...` 导入服务
- 使用 `from ml.api.schemas.stylist import ...` 导入 Schema
- chat 和 outfit 端点使用 task_worker 异步模式（来自 v1 的模式）
- analyze-body 和 guide 端点使用同步 await 模式
- 统一错误处理：使用 MLError 子类而非 HTTPException
- 服务实例使用懒加载单例模式
</action>
