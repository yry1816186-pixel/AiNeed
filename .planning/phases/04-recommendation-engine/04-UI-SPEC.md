## Interaction Specifications

### Tab Switching Flow

1. User taps tab → tab indicator animates to new position (spring)
2. MasonryFlashList data source switches to corresponding query
3. If "场合" tab → OccasionSubTabs appear with slide-down animation (200ms)
4. If leaving "场合" tab → OccasionSubTabs slide up and hide (200ms)
5. Content cross-fades (150ms) — old content fades out, new content fades in

### Swipe Feedback Flow (D-08, D-09)

1. User starts horizontal pan on card
2. At 20px translation: feedback overlay begins appearing (like/dislike)
3. At 60px translation: threshold reached — releasing will trigger action
4. Below 60px: releasing snaps card back to center (spring animation)
5. On trigger: card slides off screen (300ms timing), callback fires
6. Right swipe → `BehaviorTrackingService.track('like')` → Redis preference cache updated
7. Left swipe → `BehaviorTrackingService.track('dislike')` → Redis preference cache updated
8. Next refresh: recommendations already adjusted (D-09)

### Occasion Sub-tab Flow (D-05)

1. User taps "场合" main tab → sub-tabs slide in from top
2. Default selection: "通勤" (first item)
3. User taps different occasion → active pill switches, data reloads
4. Loading state: skeleton replaces current cards
5. Data arrives: cards fade in with stagger (50ms per card)

### Pull to Refresh

1. User pulls down → RefreshControl activates
2. Current tab's query refetches
3. On success: new data replaces old, scroll position resets to top
4. On failure: error toast appears, old data remains

### Infinite Scroll (REC-11)

1. User scrolls near bottom (threshold 0.5) → fetchNextPage triggers
2. Loading footer appears at bottom ("加载中...")
3. New cards append to existing list
4. isFetchingNextPage guard prevents duplicate requests
