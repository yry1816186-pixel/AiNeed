</objective>

<action>
Split @xuno/types from single-file to modular structure and consolidate shared types:

**Step 1: Audit current @xuno/types**
- Read `packages/types/src/index.ts` completely
- Catalog all exported types/interfaces/enums
- Compare with mobile `src/types/` to identify overlaps and gaps

**Step 2: Split @xuno/types into sub-modules**
Create domain-specific files under `packages/types/src/`:
- `auth.ts` — Auth-related types (LoginCredentials, RegisterData, TokenPair, etc.)
- `user.ts` — User types (User, UserProfile, UserPreferences, StyleProfile, etc.)
- `clothing.ts` — Clothing types (ClothingItem, ClothingCategory, ClothingFilter, etc.)
- `recommendation.ts` — Recommendation types (StyleRecommendation, OutfitPlan, etc.)
- `quiz.ts` — Quiz types (QuizQuestion, QuizAnswer, QuizResult, etc.)
- `consultant.ts` — Consultant types (Consultant, ChatMessage, etc.)
- `customization.ts` — Customization types (CustomizationRequest, DesignTemplate, etc.)
- `social.ts` — Social types (Post, Comment, Blogger, etc.)
- `order.ts` — Order types (Order, OrderItem, Payment, etc.)
- `api.ts` — API types (ApiResponse, PaginatedResponse, ApiError, etc.)
- `common.ts` — Shared enums and utility types

Update `packages/types/src/index.ts` to re-export from all sub-modules.

**Step 3: Resolve naming conflicts**
- Mobile currently uses aliases (e.g., `SharedClothingItem`) to avoid conflicts
- Extend @xuno/types types to cover mobile's additional fields
- Remove the need for aliases by making @xuno/types the canonical source

**Step 4: Move mobile-only types to shared/types/**
- `animations.ts` → `src/shared/types/animations.ts` (mobile-specific)
- `components.ts` → `src/shared/types/components.ts` (mobile-specific)
- `events.ts` → `src/shared/types/events.ts` (mobile-specific)
- `navigation.ts` → keep in `src/navigation/types.ts` (co-located with navigation)

**Step 5: Update mobile imports**
- Replace local type imports with `@xuno/types` imports where types are now shared
- Keep mobile-specific types in `src/shared/types/`
- Update `src/types/index.ts` to re-export from `@xuno/types` for backward compatibility

**Step 6: Rebuild and verify**
- `cd packages/types && pnpm build`
- `cd apps/mobile && npx tsc --noEmit`
</action>

<action>
Configure mobile app to resolve @xuno/types and @xuno/shared packages:

1. **Verify @xuno/shared re-exports from @xuno/types correctly**
   - Update `packages/shared/src/index.ts` to include all new type exports
   - Rebuild: `cd packages/shared && pnpm build`

2. **Configure TypeScript path resolution**
   Update `apps/mobile/tsconfig.json` to add paths:
   ```json
   {
     "compilerOptions": {
       "paths": {
         "@xuno/types": ["../../packages/types/src"],
         "@xuno/shared": ["../../packages/shared/src"]
       }
     }
   }
   ```

3. **Configure Metro bundler resolution** (TypeScript paths don't work in Metro)
   Update `apps/mobile/metro.config.js` to add `watchFolders` and `resolver.extraNodeModules`:
   ```javascript
   const path = require('path');
   const projectRoot = __dirname;
   const monorepoRoot = path.resolve(projectRoot, '../..');
   
   module.exports = {
     watchFolders: [monorepoRoot],
     resolver: {
       extraNodeModules: {
         '@xuno/types': path.resolve(monorepoRoot, 'packages/types/src'),
         '@xuno/shared': path.resolve(monorepoRoot, 'packages/shared/src'),
       },
     },
     // ... existing config
   };
   ```

4. **Configure Babel module resolution** (if needed)
   Check if `babel-plugin-module-resolver` is installed. If not, add it:
   ```bash
   cd apps/mobile && pnpm add -D babel-plugin-module-resolver
   ```
   Update `babel.config.js`:
   ```javascript
   plugins: [
     ['module-resolver', {
       alias: {
         '@xuno/types': '../../packages/types/src',
         '@xuno/shared': '../../packages/shared/src',
       }
     }]
   ]
   ```

5. **Verify resolution works**
   - `cd apps/mobile && npx tsc --noEmit` — TypeScript resolves paths
   - `cd apps/mobile && npx react-native start --port 8081` — Metro resolves modules

6. **Backend integration (EXPLORATORY ONLY — not a hard acceptance criterion)**
   - Check if backend can import from @xuno/types without breaking Prisma types
   - If compatible, update 1-2 backend files as proof-of-concept
   - If incompatible, document the gap for a future Phase
</action>
