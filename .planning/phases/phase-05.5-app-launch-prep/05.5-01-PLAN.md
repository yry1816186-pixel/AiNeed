# Phase 5.5 Plan 01: Backend Push Notification Infrastructure

## Objective

Build the backend push notification abstraction layer supporting FCM (Android) and APNs (iOS), create the notification template system with 4 categories, and enhance notification preference management for per-type toggling and do-not-disturb scheduling.

## Context

@05.5-CONTEXT.md
@apps/backend/src/modules/notification/services/notification.service.ts
@apps/backend/src/modules/notification/notification.module.ts
@apps/backend/src/modules/notification/notification.controller.ts

Existing infrastructure:
- NotificationService with WebSocket push, in-app CRUD, and per-user settings (email/push/inApp JSON fields)
- UserNotificationSetting Prisma model with JSON fields for email/push/inApp preferences
- NotificationType enum covering subscription, try_on, recommendation, social, system, and community types
- PrivacyService with recordConsent() for GDPR/PIPL consent tracking
- PaymentNotificationListener for payment event-driven notifications

## Tasks

### Task 1: Push Notification Abstraction Layer
<task type="auto">
<read_first>
apps/backend/src/modules/notification/services/notification.service.ts
apps/backend/src/modules/notification/notification.module.ts
apps/backend/package.json
</read_first>
<action>
Create a unified push notification service with provider abstraction:

1. Create `apps/backend/src/modules/notification/services/push-notification.service.ts`:
   - Define `PushProvider` interface with `send(token, payload)` and `sendBatch(tokens, payload)` methods
   - Implement `FcmPushProvider` using `firebase-admin` SDK (initialize with service account JSON from env)
   - Implement `ApnsPushProvider` using `apn` package (initialize with cert/key from env)
   - Create `PushNotificationService` that:
     - Auto-detects provider from device token prefix (FCM tokens vs APNs tokens)
     - Stores device tokens in a new `PushDeviceToken` table or extends User model
     - Handles token registration/deregistration endpoints
     - Has retry logic with exponential backoff (max 3 retries)
     - Logs push delivery status for debugging

2. Create `apps/backend/src/modules/notification/dto/push-notification.dto.ts`:
   - `RegisterDeviceTokenDto`: { token, platform: 'ios' | 'android', appId }
   - `SendPushDto`: { userIds, title, body, data?, category? }
   - Validation decorators

3. Add `firebase-admin` and `apn` to backend package.json dependencies (as optional peer deps with try/catch init)

4. Add Prisma model for device tokens:
   ```prisma
   model PushDeviceToken {
     id        String   @id @default(uuid())
     userId    String
     token     String
     platform  String   // 'ios' | 'android'
     appId     String?
     isActive  Boolean  @default(true)
     createdAt DateTime @default(now())
     updatedAt DateTime @updatedAt

     user User @relation(fields: [userId], references: [id], onDelete: Cascade)

     @@unique([userId, token])
     @@index([userId])
     @@index([token])
   }
   ```
</action>
<acceptance_criteria>
- PushNotificationService with FCM and APNs provider implementations
- Device token registration/deregistration API
- Graceful fallback when provider SDK not configured (log warning, skip push)
- No build errors
</acceptance_criteria>
</task>

3. Template interface:

### Task 3: Enhanced Notification Preferences & Push Integration
<task type="auto">
<read_first>
apps/backend/src/modules/notification/services/notification.service.ts
apps/backend/src/modules/notification/notification.controller.ts
</read_first>
<action>
Enhance the notification service to integrate push notifications and expand preference management:

1. Update `NotificationService.send()`:
   - After creating the DB notification and WebSocket push, check user's push preferences
   - If push enabled for the notification's category, call PushNotificationService
   - Use NotificationTemplateService to render title/body for push
   - Handle failures gracefully (log and continue)

2. Expand `PushNotificationSettings` to include per-category toggles:
   ```typescript
   interface PushNotificationSettings {
     order: boolean;           // 订单通知
     recommendation: boolean;  // 推荐通知
     community: boolean;       // 社区通知
     system: boolean;          // 系统通知
     quietHoursEnabled: boolean;
     quietHoursStart: string;  // "22:00"
     quietHoursEnd: string;    // "08:00"
   }
   ```

3. Add new controller endpoints:
   - `POST /notifications/device-token` - Register/update device token
   - `DELETE /notifications/device-token/:token` - Deregister device token
   - `GET /notifications/device-tokens` - List user's registered devices

4. Add order event listener:
   Create `apps/backend/src/modules/notification/listeners/order-event.listener.ts`:
   - Listen for order status change events (payment, shipped, delivered, cancelled)
   - Use NotificationTemplateService to render appropriate template
   - Send via NotificationService (which handles DB + WebSocket + Push)

5. Update `NotificationModule` to import and provide all new services

6. Add default push settings creation on user registration (or lazy create on first settings read)
</action>
<acceptance_criteria>

- Per-category push toggle in user settings (order/recommendation/community/system)
- Quiet hours support (don't push during configured hours)
- Device token CRUD endpoints working
- Order event listener sends templated notifications
- All new code compiles without errors
</acceptance_criteria>
</task>
