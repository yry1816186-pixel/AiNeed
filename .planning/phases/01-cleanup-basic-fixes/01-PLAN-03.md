<task id="1-03-02" type="fix">
<read_first>
- apps/backend/src/common/security/user-key.service.ts
- apps/backend/prisma/schema.prisma (User model 部分)
</read_first>
<action>
修复 user-key.service.ts 中 Prisma 可空类型导致的 TypeScript 错误。

Prisma schema 中 `encryptionKeySalt String?` 意味着类型为 `string | null`。
当前代码已有运行时 null 处理（`||` 和 `&&` 检查），但 TypeScript 可能对类型推断有问题。

具体修复步骤：

1. 运行 `cd apps/backend && npx tsc --noEmit 2>&1 | grep "user-key.service"` 确认具体错误
2. 如果错误是 `encryptionKeySalt` 可能为 null：
   - 在使用 `user.encryptionKeySalt` 的地方，确保类型守卫已正确处理
   - 第 77 行：`const salt = user.encryptionKeySalt || crypto.randomBytes(SALT_LENGTH).toString("hex");` — 已有 null 处理
   - 第 111-112 行：`if (user.encryptedDek && user.encryptionKeySalt)` — 已有 null 检查
   - 如果 TS 仍报错，添加显式类型断言：`user.encryptionKeySalt!` 或添加非空检查
3. 如果错误是 `encryptionKeySalt` 可能为 undefined（非 null）：
   - Prisma `String?` 生成 `string | null`，不是 `string | undefined`
   - 可能需要检查是否有 `strictNullChecks` 配置问题

重要：不改变任何运行时逻辑，仅修复类型注解。
</action>
<acceptance_criteria>

- user-key.service.ts 中的加密逻辑未被改变（grep 验证关键函数签名不变）
- `grep "deriveKek\|deriveDek\|encryptDek\|decryptDek" apps/backend/src/common/security/user-key.service.ts` 仍返回所有方法
</acceptance_criteria>
